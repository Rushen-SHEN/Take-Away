01 The December 2025 Agentic Inflection · 2025 年 12 月的 Agentic 拐点
Question Stephanie Zhan A couple of months ago, you said you've never felt more behind as a programmer. That's startling to hear from you, of all people. Can you help us unpack that? Was that feeling exhilarating or unsettling?
提问 Stephanie Zhan 几个月前你说，你从未像现在这样感到作为程序员如此落后。从你口中说出这话尤其令人震惊。能帮我们拆解一下吗？这种感觉是令人振奋，还是令人不安？
AndrejA mixture of both, for sure. Like many of you, I've been using agentic tools like Claude Code, Codex, and adjacent things for a while — maybe over the last year. They were very good at chunks of code, but sometimes they would mess up and you had to edit them. They were helpful.
Andrej肯定是两者兼有。和你们很多人一样，我用 Claude Code、Codex 这类 Agent 工具已经有一段时间了，大概超过一年。它们很擅长写一段段代码，但有时会搞砸，你得去改。总体是有帮助的。
agentic/əˈdʒɛntɪk/
代理式的：AI 能自主规划、调用工具、执行多步任务的特性
chunk/tʃʌŋk/
代码块：一次生成的一整段代码
Core Insight Andrej Then I would say December was a clear point. I was on a break, so I had more time. I started to notice that with the latest models, the chunks just came out fine. Then I kept asking for more and they still came out fine. I couldn't remember the last time I corrected it. I started trusting the system more and more.
核心洞见 Andrej 然后我得说，12 月是个明确的转折点。我在休假，时间更多。我开始注意到，用最新的模型，生成的代码块就是对的。我不断要更多，结果还是对的。我记不起上一次去纠正它是什么时候了。我开始越来越信任这个系统。
step change/stɛp tʃeɪndʒ/
阶跃变化：非线性、突变式的进步，而非渐进改善
AndrejI do think it was a stark transition. A lot of people experienced AI last year as a ChatGPT-adjacent thing, but you really had to look again as of December, because things changed fundamentally — especially in this agentic, coherent workflow. It really started to work.
Andrej我确实觉得这是个鲜明的转变。很多人去年把 AI 体验成「ChatGPT 那类东西」，但到了 12 月你真的得重新审视，因为有些东西发生了根本变化——尤其是在这种代理式、连贯的工作流里。它真的开始奏效了。
coherent/koʊˈhɪrənt/
连贯的：逻辑自洽、结构完整的
AndrejThat realization sent me down the rabbit hole of infinite side projects. My side-projects folder is extremely full with random things. I was coding all the time. That happened in December, and I've been looking at the repercussions since.
Andrej这个领悟把我带进了无穷无尽的副业项目的兔子洞。我的「side-projects」文件夹塞满了各种随手做的东西。我一直在写代码。这都发生在 12 月，之后我一直在观察它带来的连锁反应。
rabbit hole/ˈræbɪt hoʊl/
兔子洞：让人越钻越深、欲罢不能的探索过程
repercussion/ˌriːpərˈkʌʃən/
连锁反应：某个变化带来的后续影响
Core Insight The unit of programming changed from typing lines of code to delegating larger "macro actions": Implement this feature. Refactor this subsystem. Research this library. Set up this service. Write tests, run them, and fix failures. Compare approaches and propose a plan. The programmer is increasingly not just a code writer, but an orchestrator of agents.
核心洞见 编程的基本单元变了——从敲代码行，变成委派更大的「宏观动作」：实现这个功能、重构这个子系统、调研这个库、搭建这个服务、写测试并运行修复、比较方案并提出计划。程序员越来越不只是代码编写者，而是 Agent 的编排者。
macro action/ˈmækroʊ ˈækʃən/
宏观动作：Agent 层面的一次高层指令，内含多个子步骤
orchestrator/ˈɔːrkɪstreɪtər/
编排者：协调多个 Agent 完成复杂任务的角色
02 Software 3.0 · 软件 3.0 范式
★ Rushen's Take-Away 1Question Stephanie Zhan You've talked about LLMs as a new computer. It isn't just better software; it's a new computing paradigm. Software 1.0 was explicit rules. Software 2.0 was learned weights. Software 3.0 is this. If that is true, what does a team build differently the day they actually believe it?
提问 Stephanie Zhan 你说过 LLM 是一种新计算机。它不只是更好的软件，而是一种全新的计算范式。Software 1.0 是显式规则，Software 2.0 是学习得到的权重，Software 3.0 就是现在这个。如果这是真的，一个团队在真正相信它的那一天，会开始做哪些不一样的事？
Software 1.0
Humans write explicit code
人类写显式代码
→
Software 2.0
Datasets + neural nets
数据集 + 神经网络
→
Software 3.0
Prompts + context + tools
提示词 + 上下文 + 工具
AndrejSoftware 1.0 is writing code. Software 2.0 is programming by creating datasets and training neural networks — programming becomes arranging datasets, objectives, and architectures. Then, if you train LLMs on a sufficiently large set of tasks — and the internet contains many tasks — these models become programmable computers in a certain sense.
AndrejSoftware 1.0 是写代码。Software 2.0 是通过创建数据集、训练神经网络来编程——编程变成了安排数据集、目标函数和网络架构。接着，如果你在足够多的任务上训练 LLM——而互联网本身就包含大量任务——这些模型在某种意义上就变成了「可编程的计算机」。
weights/weɪts/
权重：神经网络中通过训练学到的参数，Software 2.0 的「程序」
objective/əbˈdʒɛktɪv/
目标函数：训练时优化的数学目标
★ Rushen's Take-Away 1Paradigm Shift Andrej Software 3.0 is about programming through prompting. What's in the context window is your lever over the interpreter, and the interpreter is the LLM. It interprets your context and performs computation in digital information space.
范式转移 Andrej Software 3.0 是通过「提示」来编程。Context Window（上下文窗口）里的内容，就是你操纵解释器的杠杆，而解释器就是 LLM。它解释你的上下文，在数字信息空间中执行计算。
context window/ˈkɒntekst ˈwɪndoʊ/
上下文窗口：LLM 一次能「看到」和处理的最大 token 量，即程序的「内存空间」
interpreter/ɪnˈtɜːrprɪtər/
解释器：逐行/逐段读取并执行指令的程序；此处 LLM 即新解释器
AndrejA few examples drove this home. When OpenClaw came out, to install it you would normally expect a shell script. But to target many platforms, shell scripts balloon and become extremely complex — you're stuck in the Software 1.0 universe of wanting to write exact code. The OpenClaw install was instead a block of text you copy-paste into your agent. It's like a little skill: copy this, give it to your agent, and it installs OpenClaw. The agent looks at your environment, performs intelligent actions, and debugs in the loop.
Andrej有几个例子让我彻底理解了这点。OpenClaw 出来时，装它你通常会预期一个 shell 脚本。但要适配许多平台，shell 脚本会膨胀、变得极其复杂——你被困在 Software 1.0 那种「想把每一步写死」的世界里。而 OpenClaw 的安装方式是一段文本，你复制粘贴给你的 Agent。它就像一个小小的 skill：复制这个、交给 Agent，它就把 OpenClaw 装好了。Agent 会查看你的环境、执行智能动作、在循环中调试。
brittle/ˈbrɪtl/
脆弱的：容易因边界情况而崩溃的
OpenClaw/ˈoʊpən klɔː/
开源 Agent 编码工具，其「复制粘贴给 Agent」的安装方式成为 Software 3.0 的标志案例
Paradigm Shift That is a different way of thinking. What is the piece of text to copy-paste into your agent? That is now part of the programming paradigm. It is less precise, but more adaptive.
范式转移 这是一种不同的思维方式。「该复制粘贴什么文本给 Agent？」——这现在成了编程范式的一部分。它精度更低，但适应性更强。
adaptive/əˈdæptɪv/
自适应的：能根据环境变化动态调整的
skill/skɪl/
技能包：一段可复用、粘贴给 Agent 的指令
AndrejAnother example is MenuGen. You sit down at a restaurant, get a menu with no pictures — I don't know what many of these dishes are. I wanted a photo of the menu to give me pictures of what those dishes might look like. So I built an app: you upload a photo, it OCRs all the titles, uses an image generator, and shows them to you. It runs on Vercel and rerenders the menu.
Andrej另一个例子是 MenuGen。你坐在餐厅，拿到一份没有图片的菜单——很多菜我根本不知道是什么。我想拍张菜单照片，让它给我这些菜大概长什么样的图片。于是我做了个 App：你上传照片，它 OCR 出所有菜名，用图像生成器出图，再展示给你。它跑在 Vercel 上，重新渲染菜单。
OCR/ˌoʊ siː ˈɑːr/
光学字符识别：从图像中提取文字的技术
Vercel/vɜːrˈsɛl/
前端应用部署平台
Core Insight Then I saw the Software 3.0 version, which blew my mind. You take the photo, give it to Gemini, and say: use Nano Banana to overlay the dishes onto the menu. It returns an image of the menu I took, but with pictures rendered into the pixels. All of MenuGen is spurious in that framing. That app shouldn't exist. Your prompt is the image, the output is an image — no need for all the app machinery in between.
核心洞见 后来我看到了 Software 3.0 版本，令我震撼。你拍照片，交给 Gemini，说：用 Nano Banana 把菜品图叠加到菜单上。它返回的是我拍的那张菜单，但菜品图片已经渲染进了像素里。在这个框架下，整个 MenuGen 都是多余的。那个 App 不该存在。你的提示就是那张图，输出也是一张图——中间那一整套 App 机器根本不需要。
multimodal/ˌmʌltiˈmoʊdl/
多模态：能同时处理文本、图像、音频等多种输入类型的模型
Nano Banana/ˈnænoʊ bəˈnænə/
Gemini 的图像生成/编辑能力代号，可 overlay 渲染到原图
spurious/ˈspjʊriəs/
多余/虚假的：本不必要、可被省去的
Core Insight People have to reframe. Don't only work in the existing paradigm and think of AI as a speedup of what exists. New things are available now. This is more general information processing that is now automatable. With my LLM knowledge bases project, you get LLMs to create wikis for your org or yourself — take messy documents, recompile them, reorder them, create something new. No classical program could do that. I keep coming back to: not only what can we do faster, but what couldn't be possible before?
核心洞见 人们必须重新框定问题。不要只在现有范式里工作，把 AI 当成对已有事物的加速。全新的东西现在就已经可用了。这是一种更普遍的信息处理，现在变得可自动化了。以我的 LLM 知识库项目为例：让 LLM 为你的组织或你个人建 wiki——把杂乱的文档拿来，重新编译、重新排序，创造出新东西。经典程序做不到这一点。我不断回到这个问题：不只是「能更快做什么」，而是「以前根本不可能、现在却很自然的是什么」？
reframe/riːˈfreɪm/
重新框定：换一套框架去看待问题
knowledge base/ˈnɒlɪdʒ beɪs/
知识库：把原始资料结构化沉淀、可查询的信息集合
03 Neural Computers · 神经计算机
Question Stephanie Zhan I love the MenuGen progression. If you extrapolate further, what is the 2026 equivalent of building websites in the 90s, mobile apps in the 2010s, or SaaS in the cloud era? What will look obvious in hindsight that is still mostly unbuilt today?
提问 Stephanie Zhan 我很喜欢 MenuGen 这个演进。如果再往前外推，2026 年的对应物会是什么——就像 90 年代建网站、2010 年代做手机 App、云时代做 SaaS 那样？有什么事回头看会觉得理所当然、但今天还基本没人建出来？
extrapolate/ɪkˈstræpəleɪt/
外推：从已知趋势推断更远的未来形态
AndrejGoing with the MenuGen example, a lot of this code shouldn't exist. The neural network should be doing most of the work.
Andrej顺着 MenuGen 的例子讲，很多这类代码根本不该存在。大部分工作应该由神经网络来做。
Paradigm Shift The extrapolation looks very weird. You could imagine completely neural computers — a device that takes raw video or audio into a neural net and uses diffusion to render a UI unique for that moment.
范式转移 这个外推看起来非常怪异。你可以设想完全的「神经计算机」——一个设备，把原始视频或音频喂进神经网络，用扩散模型即时渲染出一个专为此刻定制的 UI。
diffusion/dɪˈfjuːʒən/
扩散模型：主流的图像/视频生成方法，从噪声逐步去噪成图
AndrejIn the early days of computing, people were a little confused about whether computers would look like calculators or neural nets. In the 1950s and 60s it was not obvious which way it would go. We went down the calculator path and built classical computing.
Andrej在计算的早期，人们其实有点搞不清计算机会长得像「计算器」还是像「神经网络」。在 1950、60 年代，走哪条路并不明显。我们最终走了计算器那条路，建起了经典计算。
Paradigm Shift Neural nets currently run virtualized on existing computers. But you can imagine a flip where the neural net becomes the host process and CPUs become coprocessors — intelligence compute becomes the dominant spend of FLOPs, and tools are a historical appendage for deterministic tasks. We'll get there piece by piece.
范式转移 神经网络目前是虚拟化地跑在现有计算机上的。但你可以设想一次「翻转」：神经网络成为宿主进程，而 CPU 变成协处理器——智能计算成为 FLOPs 的主要开销，工具只是留给确定性任务的「历史遗留附件」。我们会一块一块地走到那里。
host process/hoʊst ˈprɒsɛs/
宿主进程：掌控全局、调度其他部件的主程序
coprocessor/ˈkoʊprɒsɛsər/
协处理器：辅助主处理器、承担特定计算的部件
FLOPs/flɒps/
每秒浮点运算次数：衡量算力的单位
04 Verifiability & Jagged Intelligence · 可验证性与锯齿状智能
★ Rushen's Take-Away 2Question Stephanie Zhan I'd love to talk about verifiability — the idea that AI automates faster in domains where the output can be verified. If that framework is right, what work is about to move much faster than people realize? And what professions do people think are safe, but are actually highly verifiable?
提问 Stephanie Zhan 我很想聊聊「可验证性」——也就是「在输出可被验证的领域，AI 会更快地自动化」这个想法。如果这个框架成立，哪些工作即将以超出人们预期的速度加速？又有哪些职业，人们以为很安全、实际上却高度可验证？
★ Rushen's Take-Away 2Core Insight Andrej Traditional computers automate what you can specify in code. This latest round of LLMs can automate what you can verify. When frontier labs train these LLMs in giant reinforcement learning environments with verification rewards, the models become jagged entities: they peak in verifiable domains like math and code, and stay rough where things are not in that space.
核心洞见 Andrej 传统计算机自动化「你能在代码中指定的东西」。而这一轮 LLM 能自动化「你能验证的东西」。当前沿实验室在巨大的、带验证奖励的强化学习环境里训练这些 LLM 时，模型就变成了「锯齿状」的存在：它们在数学、代码这类可验证领域达到顶峰，而在不属于这个空间的地方仍然粗糙。
verifiability/ˌverɪfaɪəˈbɪlɪti/

Andrej这有点哲学，但核心是接受一个事实：这些不是动物智能。你冲它们吼，它们不会因此做得更好或更差——它们是统计模拟的电路。这是一种心态：我在和什么打交道、什么大概会奏效、什么不会、我该如何调整？它更多是对系统保持怀疑，随时间在经验中把它摸清楚。

AndrejThe industry has to decompose workloads into sensors and actuators over the world. A sensor turns some state of the world into digital information; an actuator lets an agent change something. ∑ The Big Picture · 全景总结
Final Thesis AI is becoming a new operating layer for digital work. The scarce thing is shifting: less scarce — code generation, API recall, boilerplate, first drafts, repetitive setup. More scarce — understanding, taste, eval design, security, system boundaries, agent orchestration, domain-specific feedback loops, and knowing when the model is off the rails.
最终论点 AI 正在成为数字工作的新操作层。稀缺物在转移：不再稀缺——代码生成、API 记忆、样板代码、初稿、重复性搭建。更加稀缺——理解、品味、评估设计、安全、系统边界、Agent 编排、领域特定反馈循环，以及「知道模型何时脱轨」。
